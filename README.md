## What is ReconLens?

ReconLens is an HTTP-layer security observability tool that replaces raw scanner output with **structured, explainable intelligence**. You give it a URL or paste a raw HTTP exchange — it fetches the target, runs a deterministic rule engine across headers, fingerprints, and routing signals, then uses an LLM to translate findings into plain-language guidance with prioritized recommendations.

Designed to slot into existing SOC pipelines: ingest from triage queues, emit JSON reports, integrate with ticketing via webhook handlers.

## Features

### 🔬 Analysis Engine
- **Security header audit** — checks for CSP, HSTS, X-Frame-Options, Permissions-Policy, Referrer-Policy, and more
- **Technology fingerprinting** — detects frameworks, CDNs, server stacks, and third-party services from response signals
- **Sensitive path probing** — checks for exposed `.git/HEAD`, `.env`, `/phpmyadmin`, `/wp-admin`, `/admin`, and other high-risk endpoints
- **Endpoint classification** — categorizes the target (API, CMS, static site, SPA, etc.)
- **Security scoring** — 0–100 composite score with contextual posture label

### 🤖 AI-Powered Insights
- **Executive summary** — plain-language briefing generated by Gemini, scoped to actual findings
- **Per-header explanations** — on-demand AI explanation for any security header
- **Cookie risk analysis** — AI audit of cookie flags (Secure, HttpOnly, SameSite)
- **Vulnerability synthesis** — AI-generated vulnerability assessment from detected signals
- **Pentest assistant chat** — interactive chat widget grounded in the current scan result
- **Full report generation** — export a comprehensive AI-written report as Markdown

### 🕵️ Recon & OSINT
- **Sensitive path discovery** with severity levels (critical / high / medium / info)
- **Subdomain enumeration** via OSINT signals with one-click re-analysis
- **Client-side risk detection** — external scripts, inline event handlers, mixed content, open redirects, exposed API keys in JS
- **Secrets & leaks panel** — HTML comment analysis, hidden form fields, internal IP addresses, exposed email addresses

### 📋 Workspace
- **Persistent scan history** — all analyses stored locally, searchable by domain, technology, endpoint, or score
- **Side-by-side comparison** — select any two scans for a diff-style comparison
- **Re-scan from history** — one click to re-run any past analysis

## Getting Started

### Prerequisites

- [Node.js](https://nodejs.org/) 20+ or [Bun](https://bun.sh/) 1.3+
- A Google Gemini API key (set in `.env`)

### Installation

```bash
git clone https://github.com/sujith-himself/reconlensai.git
cd reconlensai
bun install        # or: npm install
```

### Environment

Create a `.env` file in the project root:

```env
GEMINI_API_KEY=your_gemini_api_key_here
```

> **Note:** The `.env` file is gitignored. Never commit API keys to version control.

### Development

```bash
bun run dev        # or: npm run dev
```

The app will be available at [http://localhost:3000](http://localhost:3000).

### Build

```bash
bun run build      # production bundle
bun run preview    # preview the production build locally
```

## Usage

### Analyze by URL

1. Open the **Analyze** tab
2. Enter a target URL (e.g. `https://example.com`)
3. Click **Analyze** — the engine runs a 7-step pipeline:
   - Connecting to target → Fetching headers → Downloading HTML → Probing sensitive paths → Running security rules → Generating AI insights → Preparing report

### Analyze by Raw HTTP

Paste a raw HTTP request or response directly into the text area instead of a URL. Useful for offline analysis, intercepted traffic, or replaying Burp captures.

### Reading Results

Results are organized across **9 tabs**:

| Tab | Contents |
|---|---|
| **Overview** | Security score, AI executive summary, endpoint classification, top technologies |
| **Headers** | Full security header audit with status indicators |
| **Technologies** | Detected stack with categories |
| **Recommendations** | Prioritized, AI-written remediation steps |
| **Recon** | Sensitive path probe results with severity |
| **OSINT** | Subdomain enumeration, one-click drill-down |
| **Client Risks** | External scripts, mixed content, open redirects, exposed keys |
| **Secrets & Leaks** | HTML comments, hidden fields, internal IPs, email addresses |
| **Raw Data** | Full structured JSON output from the analysis engine |

## Analysis Pipeline

```
Input (URL or Raw HTTP)
        │
        ▼
   HTTP Fetch & Header Extraction
        │
        ▼
   Security Rule Evaluation
   (headers, fingerprints, routing signals)
        │
        ▼
   Endpoint Classification
        │
        ▼
   Sensitive Path Probing
        │
        ▼
   Client-Side Risk Analysis
        │
        ▼
   OSINT Signals & Subdomain Discovery
        │
        ▼
   AI Synthesis (Gemini)
        │
        ▼
   Structured Report + Security Score
```

Each stage emits structured events rendered progressively in the UI as they resolve.

## Scoring

The security score (0–100) is a composite signal based on:

- Presence and correctness of security headers
- Exposure of sensitive paths
- Cookie flag hygiene
- Detected technology risk surface
- Client-side vulnerability signals

| Range | Posture |
|---|---|
| 85–100 | ✅ Strong security posture |
| 65–84 | ⚠️ Solid baseline — hardening steps remain |
| 0–64 | 🔴 High-risk findings detected — immediate action recommended |
